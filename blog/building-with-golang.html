Building a REST API with Go

A clean, modular approach to structuring a Go API — from routing to database integration.

Go has quickly become one of my favourite languages for building backend services. Its simplicity, fast compilation, and excellent standard library make it a joy to work with. In this post, I'll walk through how I structure a REST API project in Go.

Why Go?

Before diving into the code, here's why I reach for Go when starting a new API project:

  • Minimal boilerplate — You can build a production-ready API with just the standard library.
  • Great performance — Compiled to a single binary, low memory footprint, fast startup.
  • Excellent tooling — Built-in formatting, testing, and linting tools.
  • Concurrency primitives — Goroutines and channels make handling concurrent requests straightforward.

Project Structure

Here's the directory layout I typically use:

project/
├── cmd/
│   └── api/
│       └── main.go
├── internal/
│   ├── handler/
│   ├── model/
│   ├── repository/
│   └── service/
├── pkg/
│   └── middleware/
├── go.mod
└── go.sum

The cmd/api/main.go file is the entry point. The internal directory contains application-specific code that shouldn't be imported by external packages. The pkg directory holds reusable middleware and utilities.

Setting Up the Router

For routing, I use the net/http standard library along with Go 1.22's new pattern-matching support:

package main

import (
    "log"
    "net/http"
    "project/internal/handler"
)

func main() {
    mux := http.NewServeMux()

    mux.HandleFunc("GET /api/health", handler.HealthCheck)
    mux.HandleFunc("GET /api/items", handler.ListItems)
    mux.HandleFunc("POST /api/items", handler.CreateItem)

    server := &http.Server{
        Addr:    ":8080",
        Handler: mux,
    }

    log.Println("Server starting on :8080")
    log.Fatal(server.ListenAndServe())
}

Adding Middleware

Middleware is easy to compose. Here's a simple logging middleware:

func Logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf(
            "%s %s %s",
            r.Method, r.RequestURI, time.Since(start),
        )
    })
}

Database Integration

I use the database/sql package with PostgreSQL. Here's a simple repository pattern:

type ItemRepository struct {
    db *sql.DB
}

func (r *ItemRepository) FindAll(ctx context.Context) ([]Item, error) {
    rows, err := r.db.QueryContext(ctx, "SELECT id, name FROM items")
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var items []Item
    for rows.Next() {
        var item Item
        if err := rows.Scan(&item.ID, &item.Name); err != nil {
            return nil, err
        }
        items = append(items, item)
    }
    return items, rows.Err()
}

Key Takeaways

Go's philosophy of simplicity and explicitness makes it a great choice for building APIs. The standard library covers most needs, and the ecosystem of third-party packages is mature and well-maintained.

The most important lesson I've learned is to keep things simple. Don't abstract too early — let the project's needs drive the architecture, not the other way around.
