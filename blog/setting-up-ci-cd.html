Setting Up CI/CD with GitHub Actions

How I automated testing, linting, and deployment for my personal projects — saving time and catching bugs earlier.

Before I discovered CI/CD, my workflow was simple: write code, run tests locally, push to GitHub, and then manually deploy. It worked — until I forgot to run tests one day and pushed a broken build. That's when I decided to automate the whole thing.

What is CI/CD?

CI/CD stands for Continuous Integration and Continuous Deployment. In short:

  • CI — Every time you push code, the system automatically builds and tests it.
  • CD — If the tests pass, the system automatically deploys the changes to production.

GitHub Actions makes this incredibly easy — everything lives in a .github/workflows directory in your repository.

My First Workflow

Here's a simple workflow that runs tests on every push:

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint
      - run: npm test

Breaking this down:

  • on: — Triggers the workflow when code is pushed to main or a PR is opened.
  • jobs: — Defines what to do. Each job runs on a fresh virtual machine.
  • steps: — The individual actions: checkout code, set up Node, install deps, lint, and test.

Adding a Deploy Step

Once the tests pass, I want to deploy automatically. Here's how I deploy to GitHub Pages:

deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'

    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist

Notice the needs: test — this ensures the deploy only runs if the test job succeeds. The if condition means it only deploys from the main branch.

Secrets Management

For API keys or deployment tokens, GitHub Actions has a built-in secrets manager. Go to your repository Settings → Secrets and variables → Actions to add them. Then reference them in your workflow with ${{ secrets.MY_SECRET }}.

Never hardcode secrets in your workflow files. If you accidentally commit one, rotate it immediately and use git filter-branch to remove it from history.

What I Learned

Setting up CI/CD was one of the best productivity investments I've made. The key benefits:

  • Confidence — I know every push is tested before it reaches production.
  • Speed — Automation is faster than manual processes, especially for repetitive tasks.
  • Consistency — The same steps run every time, eliminating human error.

If you haven't set up CI/CD for your projects yet, start small — just run your tests on push. You can always add more steps later.
